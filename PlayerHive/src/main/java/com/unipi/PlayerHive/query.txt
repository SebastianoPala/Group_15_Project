/*
        1. The "Friend Recommendation" (Collaborative Filtering / Triadic Closure)
Functional-centric: Suggests new friends to a specific user based on mutual friends. If Alice is friends with Bob and Charlie, but Bob and Charlie don't know each other, the system suggests Charlie to Bob.

Graph-centric: This is a classic 2-hop traversal. It finds the immediate neighbors of an anchored user, then finds the neighbors of those neighbors, while filtering out people the anchored user is already friends with (or themselves).
        // Anchor: A specific User ID (e.g., "U123")
        MATCH (u:User {id: "U123"})-[:FRIENDS_WITH]-(mutual:User)-[:FRIENDS_WITH]-(suggested:User)
        WHERE u <> suggested
          AND NOT (u)-[:FRIENDS_WITH]-(suggested)
        RETURN suggested.name, count(mutual) as mutual_friends_count
        ORDER BY mutual_friends_count DESC
        LIMIT 5;

        2. The "Game Recommendation" (Item-Based Collaborative Filtering)
Functional-centric: Suggests games to a user based on what their friends are playing. If a user's friends are all playing "Elden Ring", the system should recommend "Elden Ring" to the user.

Graph-centric: A 2-hop traversal starting from a User, going across the FRIENDS_WITH edge to find friends, and then across the PLAYED edge to find games. It filters out games the anchored user has already played.

Cypher
// Anchor: A specific User ID (e.g., "U123")
MATCH (u:User {id: "U123"})-[:FRIENDS_WITH]-(friend:User)-[:PLAYED]->(recGame:Game)
WHERE NOT (u)-[:PLAYED]->(recGame)
RETURN recGame.title, count(friend) as friends_who_play
ORDER BY friends_who_play DESC
LIMIT 5;
Why it's efficient: Anchored query. It limits the search space to the user's immediate friends and their games.


3. The "Player Matchmaker" (Common Interests)
Functional-centric: Finds users who play the exact same games as a specific user, but who are not yet their friends. This is great for a "Find people to play with" feature.

Graph-centric: Traverses from an anchored User through PLAYED to Games, then backwards through PLAYED to find other Users. It calculates a similarity score based on the number of shared games and filters out existing friends.

Cypher
// Anchor: A specific User ID (e.g., "U123")
MATCH (u:User {id: "U123"})-[:PLAYED]->(g:Game)<-[:PLAYED]-(other:User)
WHERE u <> other
  AND NOT (u)-[:FRIENDS_WITH]-(other)
RETURN other.name, count(g) as shared_games_count, collect(g.title) as shared_games
ORDER BY shared_games_count DESC
LIMIT 5;
Why it's efficient: Anchored query. It relies on the games the user plays as the bridge, avoiding a full graph scan. Note: If a user plays a wildly popular game (like Fortnite), this node could act as a "supernode", slightly slowing down the query. You could mitigate this by adding a limit or filtering out extremely common games.


5. Efficient Global Query: "Trending Games Among Friend Groups"
Functional-centric: Finds games that are frequently played by pairs of friends. This highlights games that are inherently social or "viral" within friend groups, rather than just globally popular.

Graph-centric: This is a global query (not anchored), but it looks for a very specific triangular pattern: User A knows User B, and both play Game C.

Cypher
// No Anchor - Global pattern match
MATCH (u1:User)-[:FRIENDS_WITH]-(u2:User)
// Ensure we don't count the relationship twice (A-B and B-A)
WHERE id(u1) < id(u2)
MATCH (u1)-[:PLAYED]->(g:Game)<-[:PLAYED]-(u2)
RETURN g.title, count(*) as social_play_count
ORDER BY social_play_count DESC
LIMIT 10;
Why it's efficient (for a global query): While it's a global structural search, the WHERE id(u1) < id(u2) optimization prevents calculating the same pair twice. It's looking for a very tight, closed triangle pattern, which Neo4j's pattern matching engine handles well.



6. The "Six Degrees of Separation" (Shortest Path)
Functional-centric: Finds the shortest social path connecting two users who don't know each other. Useful for a LinkedIn-style "How you are connected to this player" feature.

Graph-centric: This uses Neo4j's built-in shortestPath() function. It traverses only the FRIENDS_WITH edges to find the minimum number of hops between two specific User nodes, bounded by a maximum depth (e.g., 6 hops) to prevent runaway queries.

Cypher
// Anchors: Two specific User IDs
MATCH (start:User {id: "U123"}), (end:User {id: "U999"})
MATCH p = shortestPath((start)-[:FRIENDS_WITH*1..6]-(end))
RETURN p, length(p) AS degrees_of_separation;
Why it's efficient: Instead of exhaustively searching every possible path between the two users, shortestPath() uses a bidirectional breadth-first search, stopping the moment the two search frontiers intersect.



Functional-centric: Finds the user whose gaming library most closely matches yours, regardless of whether you are friends. This is pure similarity scoring, perfect for finding a "Gaming Twin."

Graph-centric: It calculates the Jaccard Similarity Index (Intersection over Union). It finds users who share at least one game, counts the shared games (Intersection), calculates the total unique games played by both (Union), and divides the two.

Cypher
// Anchor: A specific User ID
MATCH (u1:User {id: "U123"})-[:PLAYED]->(g:Game)<-[:PLAYED]-(u2:User)
WITH u1, u2, count(g) AS shared_games

// Count User 1's total games
MATCH (u1)-[:PLAYED]->(g1:Game)
WITH u1, u2, shared_games, count(g1) AS u1_total

// Count User 2's total games
MATCH (u2)-[:PLAYED]->(g2:Game)
WITH u2, shared_games, u1_total, count(g2) AS u2_total

// Jaccard = Intersection / (Total A + Total B - Intersection)
WITH u2, shared_games, (u1_total + u2_total - shared_games) AS union_games
RETURN u2.name, toFloat(shared_games) / union_games AS jaccard_similarity
ORDER BY jaccard_similarity DESC
LIMIT 5;
Why it's efficient: It anchors on the target user and only evaluates other users who have at least one game in common, completely ignoring users with zero overlap.


8. The "Hidden Gem" Recommendation (Inverse Popularity)
Functional-centric: Suggests games that are played by a user's friends, but are rarely played by the general public. This highlights niche, cult-classic games within a friend group rather than just recommending the most popular global hits.

Graph-centric: A 2-hop traversal to find friends' games, combined with a global aggregation on those specific games to count their total player base. We then filter out games that exceed a "popularity threshold."

Cypher
// Anchor: A specific User ID
MATCH (u:User {id: "U123"})-[:FRIENDS_WITH]-(friend)-[:PLAYED]->(game:Game)
WHERE NOT (u)-[:PLAYED]->(game)
WITH game, count(DISTINCT friend) AS friends_playing

// Check global popularity of those specific games
MATCH (game)<-[:PLAYED]-(global_player)
WITH game, friends_playing, count(global_player) AS global_popularity
WHERE global_popularity < 1000 // The "Niche" threshold

RETURN game.title, friends_playing, global_popularity
ORDER BY friends_playing DESC, global_popularity ASC
LIMIT 5;
Why it's efficient: It aggressively filters the dataset early. Instead of counting the global popularity of every game in the database, it only counts the global popularity of the specific subset of games the user's friends are playing.
